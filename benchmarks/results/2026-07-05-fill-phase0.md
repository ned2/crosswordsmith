# Fill performance campaign — Phase 0 results (2026-07-05)

Benchmark build-out for `fill` (Flavour-B dictionary fill of a blocked stock
grid, `prolog/crosswordsmith/fill.pl`). This is the FIRST measurement of fill's
performance. Companion: `docs/plans/fill-perf-campaign.md` (Phase 0). Machine:
`lana x86_64`, SWI-Prolog 10.1.10. All inference counts are deterministic and
machine-independent; wall/rss are host-specific and reporting-only.

## What was built

- **Dictionaries** (`fixtures/dict/`): ENABLE (172,823 words, public domain,
  sha-frozen) as the primary rung dictionary + seeded-shuffle subsets
  (10k/25k/50k, deterministic via `gen_fill_dict.py`, seed 20260705). ASCII,
  lowercase, LF. sha256 recorded in the README; regeneration is byte-identical.
- **Stock grids** (`fixtures/fill_grid_{04,05,09,11,13,15,17,21}a.json`): all 8
  PASS `stockgrid_validate` (required 9/15/21 + optional ladder knobs). Built
  short-word (max light length 5) so they complete against ENABLE (see thesis).
- **Seed fixture** (`fixtures/fill_seed_11a.json`): 3 pins (CYANO/TOMMY/READD)
  from grid 11a's own unseeded solution; distinct later-alphabet words so the
  search does not re-place a pinned answer (the duplicate-answer trap, below).
- **Runner** (`benchmarks/run_fill.pl` + `fill_subjects.pl` + `fill_workloads.pl`):
  four attribution buckets per rung — command / dict_load / grid / search — with
  fresh slots rebuilt outside the timed search goal each sample.
- **Ratchet** (`benchmarks/check_fill_baseline.pl` + `fill_baseline.json` +
  `fill_history.jsonl`): forked from the arrange ratchet. search_inf gates at
  0.5%; load_inf + grid_inf reported-only until Phase 3 decides.
- **Identity oracle** (`benchmarks/check_fill_identity.sh` +
  `fill_identity.sha256`): CLI stdout sha256 per rung — Phase 2's byte-identity
  gate. 11/11 verified.
- **Scale golden** (`tests/golden/fill_15_bench.json`): a 15x15 full-ENABLE fill,
  byte-stable across 3 runs, wired into `run_tests.sh`.

**dict_load exceeds 50% of end-to-end latency on 10 of 11 rungs** — 75–84% on
every full-ENABLE rung, 64% on sq04_50k, and 58% even on the top search rung
(g09_full, 34.9M search inferences). Only `g17_50k` is search-dominated (search
52.6%, load 35.7%). This directly fires the campaign plan's conditional: *"If
P-F1 shows load_dict > ~50% of end-to-end latency at realistic scale, Phase 3
outranks Phase 2 for perceived performance."* On this machine, for realistic
grids against a real dictionary, **the dominant latency is loading the
dictionary, not searching it.**

Two corollaries for where the cost lives (first measurement of fill):
- `dict_load` is a flat ~3.25s wall / 26.6M inferences on full ENABLE,
  independent of the grid — the every-invocation startup tax (read + normalize
  + full positional index over all lengths, `fill.pl:114`). It is the single
  biggest perceived-latency lever.
- `grid` (`fill_grid/4`) is negligible everywhere (≤0.11% wall, ≤111k inf).
- `rest` (11–21%, ~0.6–0.9s) is SWI VM boot + `load.pl` consult + emit — a
  fixed process tax that also dwarfs search on light rungs.
- Within `search`, cost tracks fan-out × depth as expected; g17_50k is the one
  rung where search truly dominates (a 17x17 interlock against a mid-size dict).

## Reproduction + warmup evidence

**search_inf AND load_inf reproduce to +0.00% across 3 independent full-ladder
runs** (all `--heavy`), and against the recorded baseline:

- Run 1 — `run_fill --heavy --iterations 1 --warmup 1` (repro_A): all 11 rungs
  search_inf + load_inf identical to `fill_baseline.json`.
- Run 2 — `make bench-fill-check BENCH_ARGS=--heavy` (manifest iterations 2–3):
  every gated rung +0.00%, RESULT PASS (exit 0).
- Run 3 — `run_fill --heavy --iterations 1 --warmup 1` (repro_B): identical.

The counts are independent of iteration count (a warm sample is exact), so the
it=1 runs match the it=2/3 baseline exactly.

**Cold/warm (first-call JIT indexing):**
- `load_dict`: enable_10k cold 1,605,524 / warm 1,603,141 (Δ 2,383); full ENABLE
  cold == warm 26,602,930 (clauses already first-arg-indexed by the first big
  load). The orchestrator's reference counts (10k = 1,605,524; ENABLE =
  26,602,930) reproduced EXACTLY → the harness does not perturb the count.
- `fill_attempt` (search): a **fixed 319-inference** cold/warm delta, identical
  on sq04 / g21 / g09 (a one-time first-call JIT cost, independent of tree
  size). Relative size: 0.06% on the smallest rung, <0.001% on the largest.
  Arrange's first-call delta was ~7.6k; fill's is far smaller. Because every
  rung records with `warmup≥1`, both the baseline and `bench-fill-check`
  measure the WARM count, so the gate compares warm-to-warm (zero delta).

**Fresh-slots determinism (the destructive-unification trap):** `fill_attempt`
unifies cell vars in place, so slots must be rebuilt outside the timed goal every
sample. Verified: warm search calls with fresh slots are bit-identical (sq04:
529,275 on calls 2/3/4); reusing slots would return garbage on call 2+.

## Envelope findings (non-completing regions)

The completable-FAST regime is NARROW and dictionary-content-dependent (the
plan's central thesis, confirmed). The ladder is deliberately short-word (max
light length 5) so the first greedy MRV path succeeds with shallow backtracking
against an abundant dictionary. Outside that regime, fill saturates:

- **Curated UK blocked grids** (`grids/blocked_13a/13b/15a.json`, full-length
  lights len 13–15) against full ENABLE: do NOT complete. Self-measured probe —
  `blocked_15a` (15x15, 52 slots, max light length 15) exhausted a **2e8**
  budget (~6x the top completing rung g09_full) returning `not_proven`
  (search_inf = 200,000,021, i.e. the budget cap). The full-length lights have
  too few candidates and over-constrain the interlock — the heavy-tail regime
  the campaign thesis predicts.
- **Open word-squares ≥6×6** against ENABLE: over-constrained interlock, do not
  complete within budget (triage; excluded from the ladder). The 5x5 open square
  completes (sq05_full, 1.46M) and the 4x4 completes even against the 50k subset
  (sq04_50k, 7.74M, a near-phase-transition rung kept because it is comfortably
  below saturation and byte-frozen).

The heavy tail's own top rung, g09_full (a 9x9 with dense crossings against full
ENABLE), is the hardest COMPLETING instance found at 34.9M — 57x under the 2e9
bench budget. Pushing toward the plan's ~100M ceiling meant crossing into the
non-completing regime, so the ladder ceiling is honest at ~35M, not forced.

## Rejected rung candidates (with numbers)

| candidate | why rejected |
|-----------|--------------|
| Curated blocked 13a/13b/15a × ENABLE | non-completing (envelope; full-length lights over-constrain) |
| Open squares ≥6×6 × ENABLE | non-completing (interlock saturates before a solution) |
| Seeding an early-alphabet word (e.g. AAH) | CLI exit 1: a pinned answer is EXEMPT from the no-duplicate rule (`fill.pl:196` guards searched slots only), so seeding a word the search would place anyway (AAH is every open 3-slot's first candidate) makes the search re-place it and `emit` throws `unique_key_pairs`. Fixed by seeding distinctive later-alphabet words from the rung's own solution (CYANO/TOMMY/READD). |
| Synthetic random dictionaries as primary | inter-complete far too rarely (what saturated the red-team 5x5 probe); real ENABLE completes readily. Synthetic subsets kept only as a size-scaling knob. |

## Design decisions

- **Real dictionary primary, synthetic subsets as size knob** (plan [RT]) —
  crossing-completion structure is what makes fill hard; random words don't
  model it.
- **Fork the ratchet, don't parameterize** — `check_fill_baseline.pl` is
  independent of arrange's; it gates search_inf and REPORTS load_inf (the two
  layers fill needs), with the arrange new-rung silent-drop fix carried.
- **Four buckets, grid as its own bucket** — never silently folded into "rest".
- **`--budget` CLI hook DEFERRED** (orchestrator decision) — the bench drives
  the budget-explicit `fill_attempt/8` directly at 2e9; the CLI's shipped 800M
  default still completes every rung (top rung 34.9M ≪ 800M), so the goldens and
  identity manifest use the shipped CLI unchanged.

## Candidate P-F1 focus areas (where to point the instrumentation probe)

1. **`load_dict` is the perceived-latency lever, not search** — 26.6M inf /
   ~3.25s, >50% of end-to-end on almost every rung. If Phase 3 (startup + WASM)
   is scheduled by perceived latency, it likely OUTRANKS Phase 2's per-node
   search work. The index build (`build_index/2`) over all lengths — including
   lengths the grid lacks — is the obvious first target (plan Phase 3).
2. **Within search**, the plan's ranked gaps (candidate materialization
   `candidates/4`, `ord_intersection` counting, non-incremental `select_mrv`)
   remain the hypotheses; g17_50k (search-dominated) and g09_full (deepest tree)
   are the rungs where a search win would show. P-F1 should split search into
   materialization vs counting vs unification vs used-list scans on these rungs.
3. **grid derivation is a non-issue** (≤0.11%): do not spend Phase 2 effort
   there.
