# SWI-Prolog Code Audit — Findings

This is a **read-only** audit of the SWI-Prolog crossword generator (`crossword.pl`, `quality.pl`,
and the Tier-2/Tier-3 test, benchmark, and fixture files) generated by a multi-agent workflow on
2026-06-22. Every idiom / footgun / "use X instead" claim is grounded in the **version-matched
manual** under `docs/reference/swi-manual/` (SWI-Prolog 10.0.2, the exact version this repo runs);
claims the manual does not support were dropped. No source file was modified.

How to read each finding:

- **severity** — `high` > `med` > `low` > `nit`. Reflects the verifier consensus, which often
  *downgraded* a finder's original rating; downgrades are noted in-line.
- **behaviour** — `preserving` (cannot change observable output: the 66 plunit tests, the golden
  file `tests/golden/grid_17_topleft_across.txt`, and the JSON input/output contract are untouched)
  vs `risk-HIGH` (a fix *could* change observable output and must be golden/diff-checked).
- **perf** — `none` / `likely-better` / `likely-worse` / `needs-benchmark`. The portable metric is
  INFERENCES (per `docs/experiments.md`). Per the perf fence, elegance-for-speed trades are **not**
  auto-recommended and are listed separately under NEEDS-BENCHMARK.

## Executive summary

### Counts (final surviving findings, by consensus severity)

The audit produced 26 candidate findings. After verifier consensus, several collapsed to
*drop-level* (real-but-trivial, no actionable defect: F001, F002, F017, F018, F021, F022). The
remaining surviving, actionable findings break down as:

| severity | count | ids |
|----------|-------|-----|
| high     | 1     | F016 |
| med      | 1     | F026 |
| low      | 3     | F002b, F005, F010 (F006/F009 hover low↔nit) |
| nit      | ~14   | F003, F004, F007, F008, F011, F012, F013, F014, F015, F019, F020, F023, F024, F025 |
| drop-level | 6   | F001, F002, F017, F018, F021, F022 |

### Top quick wins (Med+, behaviour=preserving, perf=none/likely-better — safe to recommend)

- **F005** (low) — `all_crossword/5` counts solutions with `length(Sols,Num), findall(...)` which
  enumerates list skeletons and re-runs the entire search N+1 times. Replace with
  `aggregate_all(count, Goal, Num)`. Behaviour-preserving (untested `--all`/count path), perf
  likely-better. The single clearly-safe-and-beneficial recommend in the set.
- **F009** (low) — dead predicates `direction/1`, `head/2`, `length_sort/2` are never called.
  Pure removal; behaviour-preserving, perf=none.
- **F012** (nit) — comment in `quality.pl` cites `quality_weights/2` but the predicate is
  `quality_weights/3`. Comment-only; behaviour-preserving, perf=none.
- **F026** (med) is behaviour-preserving with perf=none **but** lives behind a footgun caveat — see
  HIGH / behaviour-risk note below; it is a recommend for the benchmark harness, not the solver.

(The remaining nits — F003, F004, F008, F011, F013, F014, F019, F020, F023, F024, F025 — are all
behaviour-preserving/perf-none cosmetic alignments and may be folded into the next cleanup pass, but
none rises to a "win" worth its own change.)

### HIGH / behaviour-risk (called out explicitly)

- **F016 — HIGH, CHALLENGE, behaviour=risk-HIGH.** `quality.pl`'s `next_move/5` builds candidates
  with `findall(... cand(Entry,...) ...)`, which **copies** the word `Entry`. `apply_move/6` then
  calls the `==`-based `remove_x/3` on that copy; because a `.pl`-fixture entry `[Answer, _{...}]`
  carries an **unbound dict tag** (non-ground), the copy is `\==` the original, nothing is removed,
  and `greedy_loop/6` recurses on identical state **forever**. This is the *exact* E5 failure class
  the author already fixed in `crossword.pl` and correctly applied to quality.pl's *seed* path — but
  not to the move loop. Reproduced end-to-end: the shipped fixture `fixtures/bundled_17_clues.pl`
  (and any `.pl` `_{...}` input) hangs `--quality`; bare-answer fixtures and ground-tag JSON inputs
  are masked, which is why the suite (which has **zero** quality-engine coverage) stays green. The
  fix (remove by the ground answer atom, e.g. `selectchk([Ans|_], Remaining, Rest)`) turns a hang
  into output, so it is risk-HIGH by definition and `perf=needs-benchmark` (a hang has no baseline).
- **F026 — MED, behaviour=preserving but a footgun.** `read_loop/2` in `benchmarks/run_matrix.pl`
  (and the identical copy in `start_sensitivity.pl`) silently unifies `Words=[]` when a fixture has
  no `clues/1` term; an empty word list "solves" trivially and is recorded as a valid measured cell,
  silently corrupting the experiment batch. `run_benchmarks.pl` already throws
  `fixture_missing_clues/1` for the same case.
- **F002b / F015 (low/nit) — behaviour=risk-HIGH** without being severity-high: both can perturb a
  random/heuristic ordering (the `--shuffle` permutation stream; the quality seed-selection
  ordering on multi-word answers) and so must be golden/diff-checked rather than assumed safe.

### NEEDS-BENCHMARK (elegance-for-speed trades — NOT recommended without measurement)

These trade clarity for inference cost and are explicitly excluded from the recommend list:

> **Benchmarked 2026-06-23** (real suite, SWI 10.0.2) — see
> `docs/experiments.md` § "Audit micro-optimization benchmarks". Outcomes recorded
> inline below. The benchmark overturned the priors on F007 and F010; **F010 and
> F007 were then adopted** (2026-06-23 result batches), with F006 left as-is.

- **F006** (low) — factoring the 4× strip-spaces idiom into an `exclude(==(' '), ...)` helper. Verifiers
  measured `exclude/3` (meta-call) at ~1.4× more inferences than the C-builtin `delete/3` on the MRV
  hot path → **perf=likely-worse**. Also note the proposed helper would *diverge* from `quality.pl`'s
  existing `delete/3`-based `word_letters/3`, not mirror it.
  **MEASURED → REJECTED:** `exclude/3` is worse on all 28 cells (+0.09% to +4.36%, worst on `mrv_inc`);
  a delete-based DRY helper is also worse on all 28 (+0.02% to +0.77%, per-call overhead). Keep the
  inline `delete/3`s. (Audit confirmed.)
- **F007** (nit) — `mrv_count/8` `findall(t,...)+length` → `aggregate_all(count,...)`. Warm cost is
  noise at cap 2, but on the unbounded `mrv` strategy verifiers measured ~60% more inferences →
  **perf=likely-worse on the mrv/unbounded path**.
  **MEASURED → prior REFUTED; ADOPTED 2026-06-23:** marginally *faster* (mrv/mrv_capped/mrv_inc cells
  −0.09% to −0.37%, 0 worse; baseline cells byte-identical). A readability win at worst perf-neutral,
  not a +60% regression. See `docs/experiments.md` 2026-06-23 F007 result batch.
- **F010** (low) — replacing the hand-rolled `position/3`/`x_position/4` with `nth1/3` (plus deleting
  a dead `:- false` clause). On the `find_intersecting_word/6` hot path → **perf=needs-benchmark**
  (microbench showed `nth1` marginally cheaper, but measure on the real suite before adopting).
  **MEASURED → WIN (bigger than guessed); ADOPTED 2026-06-23:** 26/28 cells fewer inferences, 0 worse;
  typical −2% to −3%, −4.79% (~21.7 M) on the hard baseline `benchmark_16_dense` cell.
  `position/3`/`x_position/4` removed (with their 3 unit tests; suite now 81 plunit + golden, all green).
  See `docs/experiments.md` 2026-06-23 result batch.
- **F016** (high) — the fix's `perf=needs-benchmark` (no baseline; the engine currently never
  completes on affected inputs). Listed here for completeness, but it is a *correctness* fix and the
  current behaviour is non-terminating, so any termination is strictly an improvement.

**F001** | lens=b | `crossword.pl:118,122–125,128,131,134,179–181` | severity=nit (consensus: **drop-level**) | option-access-idiom

- **Title:** Option values read with `memberchk/2` rather than `library(option)` `option/2`.
- **Current:** `memberchk(input(InputFile), Opts)`, `memberchk(out(OutFile), Opts)`,
  `memberchk(strategy(Strategy), Opts)`, `memberchk(quality(true), Opts)`, … (~10 sites).
- **Issue:** `opt_parse/4` returns a `Name(Value)` list with every key present at its default;
  `option/2` is the documented accessor for exactly that shape. `memberchk` conflates "is this flag
  literally `true`" with "fetch this flag's value."
- **Proposal:** Prefer `option(input(F), Opts)` for the value-extracting reads; leave the
  `build_floors` `(->;)` mapping (which `option/2` cannot express) as-is. Optional.
- **manual_cite:** `swi-manual/option.md` (option/2, "list of Name(Value)"); `swi-manual/optparse.md`
  (opt_parse/4 returns Name(Value) list with defaults prepended).
- **behaviour:** preserving | **perf:** none | **confidence:** high (cites accurate).
- **Disposition:** All three verifiers DROP. The manual offers `option/2` as a convenience, never
  calls `memberchk`-for-options wrong or non-idiomatic; the value-binding idiom is valid and the
  current style is internally consistent. Real but below the actionable bar.
- **Steelman:** `memberchk/2` is a zero-import core builtin; every option carries a default so it is
  always present, making the semantics equivalent. Uniform style across `run/2` and `build_floors/2`.

**F002** | lens=b | `crossword.pl:149–151,173–174,240–241,254,278,287,292,306–307` | severity=nit (consensus: **drop-level**) | error-term-convention

- **Title:** Custom atoms in `error(Formal,_)` where standard ISO formal terms fit.
- **Current:** `throw(error(unknown_strategy(S), _))` and `json_invalid_answer/1`,
  `unsupported_clue_file/2`, `missing_input_file`, `json_no_clues_array`, etc.
- **Issue:** `unknown_strategy(S)` maps onto `domain_error(crossword_strategy, S)`;
  a non-string answer onto `type_error(string, RawAnswer)`. Generic tooling that pattern-matches the
  ISO vocabulary won't recognise the app-specific atoms.
- **Proposal:** Where a standard term fits, use `library(error)` helpers and render via
  `prolog:error_message//1`; keep genuinely app-specific events as custom terms.
- **manual_cite:** `swi-manual/error.md` (type_error/2 L15, domain_error/2 L22–23, existence_error L25–29).
- **behaviour:** **risk-HIGH** | **perf:** none | **confidence:** med.
- **Disposition:** Downgrade/not-real (consensus drop). The manual presents `*_error` helpers as
  *conveniences* (error.md:13), never mandating them over custom terms; custom terms paired with
  `prolog:error_message//1` are a supported idiom (error.md:11). Decisively, ~8 plunit tests assert
  on the exact custom formal terms (`tests/crossword.plt:167,265,268,271,274,277,286,336`), so a swap
  is observable — and the author already uses ISO terms where they genuinely fit (`existence_error`
  for missing files), proving a deliberate, discriminating design.
- **Steelman:** Each custom term has a dedicated `error_message//1` renderer producing friendlier CLI
  text; swapping would surface SWI's generic wording and break the asserting tests.

**F002b** | lens=a | `crossword.pl:194–199,988–1000` | severity=**low** | stdlib-predicate (randomisation)

- **Title:** Hand-rolled `shuffle/2` + `choose/2` reimplements `random_permutation/2`.
- **Current:** `shuffle([], []) :- !.` / `shuffle(List,[E|Rest]) :- choose(List,E), delete(List,E,New), shuffle(New,Rest).`
  with `choose(List,Elt) :- length(List,L), Index is random(L), nth0(Index,List,Elt).`
- **Issue:** Reimplements `random_permutation/2` and removes the chosen element with **deprecated**
  `delete/3` (matches by *unification*, removes ALL matches — finder said `==`; verifiers corrected
  this to `\+ Elem \= H`, which *widens* the footgun). Current callers (unique answers; four distinct
  start locs) have no duplicates so it is correct today, but it is O(n²) and a latent duplicate-drop hazard.
- **Proposal:** `shuffle(List, Shuffled) :- random_permutation(List, Shuffled).` and delete `choose/2`
  (requires adding `:- use_module(library(random))`, not currently imported).
- **manual_cite:** `swi-manual/random.md` (random_permutation/2); `swi-manual/lists.md:59–66` (delete/3 deprecated).
- **behaviour:** **risk-HIGH** (per-seed `--shuffle` layout changes under a different RNG consumption
  pattern; no test pins a seed-specific permutation — `shuffle_is_permutation` checks the multiset
  only) | **perf:** likely-better (2/3) / none (1 verifier: lists are tiny — Locs is 4 elements) →
  **needs-measurement** | **confidence:** high.
- **Disposition:** REAL, low. Manual-backed; behaviour-preserving for the protected surface (the one
  shuffle test checks multiset, golden is the deterministic top-left path, JSON is RNG-independent).
- **Steelman:** `shuffle/2` is self-contained and already random; swapping changes the specific
  per-seed layout, an observable change in the random stream.

**F004** | lens=c | `crossword.pl:297–312,754–756` | severity=nit | undocumented-hook

- **Title:** Custom CLI errors render through `prolog:error_message//1`, a hook **not documented** in
  the SWI 10.0.2 manual (the documented extension hook is `prolog:message//1`).
- **Current:** `:- multifile prolog:error_message//1.` + 8 clauses translating custom `error/2` formals.
- **Issue:** Grepping the entire vendored manual for `error_message` returns **zero** files. The
  documented hook is `prolog:message//1` (printmsg.md §4.11.1). `prolog:error_message//1` is a real,

- **Proposal:** Leave as-is (it works; changing it is behaviour-risk). Record that the *supported*
  equivalent is `prolog:message(error(Formal,_)) --> [...]`. Do not migrate blindly.
- **manual_cite:** `swi-manual/printmsg.md` §4.11.1 (prolog:message//1, L169–176); grep confirms

  pinned version (the point of vendoring the manual); the proposal recommends *no code change*; and

  thrown `error/2` term, not rendered text. Informational documentation note only.
  See also the cleared note confirming `prolog:error_message//1` is the *correct* current mechanism.
- **Steelman:** A long-standing, widely-used SWI hook, arguably more ergonomic than `prolog:message//1`

**F005** | lens=b | `crossword.pl:332–334` | severity=**low** (quick win) | non-idiomatic-aggregation

- **Title:** `all_crossword/5` measures-then-generates; `aggregate_all(count,...)` is the idiom.
- **Current:** `all_crossword(...,Num) :- length(Sols, Num), findall(Grid, find_crossword(...), Sols).`
- **Issue:** `length(Sols, Num)` runs first with both args unbound, enumerating list skeletons
  `[],[_],[_,_],…`; `findall` re-runs the **entire** search on each skeleton mismatch — verifiers
  instrumented N+1 full searches (a 3-solution goal ran 4 findalls / 12 solutions). On backtracking
  the predicate also loops forever generating ever-longer skeletons.
- **Proposal:** `aggregate_all(count, find_crossword(...,_,_), Num)`.
- **manual_cite:** `swi-manual/aggregate.md` (count = sum(1), "processed incrementally … constant
  memory", L32–33, 69).
- **behaviour:** preserving | **perf:** likely-better | **confidence:** high.
- **Disposition:** REAL (3/3). The only caller `count_solutions/4` takes the first solution and
  `writeln(Num)`s it, so the count is identical and the leftover infinite choicepoint is never
  exercised. No plunit/golden/JSON coverage of the `--all`/count path. Marginally faster (constant
  memory, no N+1 length backtracks), but the dominant search cost is unchanged.
- **Steelman:** `findall+length` is a known idiom; `aggregate_all` is itself findall-based so not
  asymptotically cheaper.

**F006** | lens=a | `crossword.pl:366–368,433–435,475–477,538–540` | severity=low/nit | stdlib-predicate / DRY

- **Title:** Strip-spaces idiom uses deprecated `delete/3`, duplicated at 4 sites.
- **Current:** `atom_chars(Word, Letters), delete(Letters, ' ', Letters2), length(Letters2, WLen)`

- **manual_cite:** `swi-manual/lists.md:59–66` (delete/3 deprecated); `swi-manual/apply.md:30` (exclude/3).
- **behaviour:** preserving (verified `exclude(==(' '))` ≡ `delete(' ')` byte-for-byte on ground char
  lists) | **perf:** **likely-worse / needs-benchmark** (exclude/3 is a meta-call, ~1.4× inferences on
  the MRV hot path vs the C-builtin delete/3) | **confidence:** high.
- **Disposition:** Downgrade to low/nit (3/3). Real DRY + deprecated-builtin observation, but the
  *specific* `exclude/3` rewrite is a perf regression on a hot path and would diverge from
  `quality.pl`'s own `delete/3`-based helper. **See NEEDS-BENCHMARK.** If consolidated at all, prefer
  keeping `delete/3` to match the existing helpers.
- **Steelman:** `delete/3` on ground char atoms is correct and fast; the sites are deliberately
  self-contained on the hottest path.

**F008** | lens=a | `crossword.pl:445–446` | severity=nit | stdlib-predicate / spurious-choicepoint

- **Title:** `capped/2` hand-codes the unbounded case that `limit/2` provides via `infinite`, and

- **manual_cite:** `swi-manual/solutionsequences.md` (limit/2: "infinite … equivalent to call/1", L56/61).
- **behaviour:** preserving | **perf:** none (the sole caller `mrv_count/8` wraps `capped/2` in
  `findall/3`, which enumerates the choicepoint away) | **confidence:** high.
- **Disposition:** Downgrade to nit (3/3). Both claims reproduced, but zero observable effect; affects
  only the non-default `mrv`/`mrv_capped` strategies.
- **Steelman:** The two-clause form is self-documenting, cut-free, with an `integer/1` guard, and the
  `unbounded` atom matches the `mrv_cap` vocabulary.

**F011** | lens=b | `crossword.pl:988–992` | severity=nit | deprecated-builtin / delete-all-vs-one

- **Title:** `shuffle/2` removes the chosen element with deprecated `delete/3` (unify-match, ALL
  occurrences) instead of `selectchk/3`.
- **Current:** `shuffle(List,[E|Rest]) :- choose(List,E), delete(List,E,New), shuffle(New,Rest).`
- **Issue:** `choose/2` picks one element by index; `delete/3` then removes *every* unifying element.
