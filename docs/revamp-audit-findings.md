# crosswordsmith Revamp Audit — Findings

A code review of the **revamp** code — the new Flavour-A/B surfaces added on top of
the original generator: `arrange.pl`, `fill.pl`, `lint.pl`, `export.pl`,
`stockgrid.pl`, and the `crosswordsmith` CLI (plus the revamp's changes to
`crossword.pl`/`quality.pl`). Generated by a multi-agent finder → adversarial-verifier
→ synthesis workflow on **2026-06-30**, against the design spec
([`design-spec.md`](./design-spec.md)) as the normative contract.

This is the companion to the earlier [`prolog-audit-findings.md`](./prolog-audit-findings.md)
(closed 2026-06-24), which swept the *pre-revamp* `crossword.pl`/`quality.pl`. Findings
already dispositioned there (F001–F026) are out of scope here.

## How to read each finding

- **severity** — `high` (wrong output / crash / hang / AC-violation on realistic input)
  > `med` (edge-case-wrong or a real contract gap) > `low` (latent / cosmetic-but-real)
  > `nit` (style). Reflects the verifier consensus, which downgraded some finder ratings.
- **dimension** — `correctness` / `spec-conformance` / `test-gap` / `doc-drift` /
  `contract` / `footgun`.
- **verdict** — `CONFIRMED` (reproduced or directly code-grounded) /
  `PLAUSIBLE` (likely real, not fully reproduced). 18 candidates were raised; 2 were
  refuted in verification; the 16 below survived.
- **status** — remediation state (see the log at the bottom).

## Method

Seven review lanes ran in parallel — spec-conformance (AC-by-AC), `arrange.pl`,
`fill.pl`, `lint.pl`+`stockgrid.pl`, `export.pl`+JSON contract, CLI+cross-cutting
invariants, and tests+docs integrity. Every candidate finding was handed to an
independent skeptic that tried to **refute** it (reading the cited code and callers,
grounding Prolog-idiom claims in the version-matched SWI 10.0.2 manual under
`docs/reference/swi-manual/`, and reproducing with `swipl`/the CLI where cheap), then a
lead reviewer deduped and ranked the survivors. Ground truth at audit time: the full
suite is green — **152 plunit + 8 CLI goldens**.

## Executive summary

The revamp is broadly functional and the suite genuinely passes, but `STATUS.md`'s
"every component built, all ACs pass" is **not** fully justified:

- **Two `high` correctness defects** silently corrupt output on common input classes:
  hyphenated answers place a literal `-` grid cell (R1), and a `fill` seed word absent
  from the dictionary makes the engine reject a fillable grid while misreporting the
  user's own pin as unfillable (R2).
- **One confirmed determinism gap** (R3) breaks the load-bearing emit→re-ingest
  byte-identity property `arrange` advertises (`AC-EMIT-2`).
- **A LOCKED §7.2 MUST is unimplemented** and contradicts §5 (R8 — `--check-target`).
- Most tellingly, **several whole spec outcome-branches have zero test coverage** —
  `not_proven` for `arrange` (R4) and `fill` (R11), and the `AC-LINT-2`
  FAIL→non-zero-exit mapping (R12) — the exact F016-class blind spot the green suite
  hides. A regression in any of them would pass green.

## High

### R1 — Hyphenated answers place the literal `-` as a grid cell
`crossword.pl:270` (and `:337`, `:381`, `:443`/`entry_letters`; `quality.pl:60`/`word_letters`) ·
dimension=correctness · AC-ENUM-1, AC-ARR-3 · **CONFIRMED**

Placement strips spaces (`delete(Letters,' ',_)`) but **not hyphens**, so `WELL-BEING`
is placed as a **10-cell** run for a 9-letter answer, with a literal `{"letter":"-"}`
cell. `export --to exolve` then prints an invalid grid row (`-` is neither letter nor
block) and the ipuz `solution` contains `-`; any crossing word landing on the hyphen
column inherits an unsatisfiable `-` fixed letter. No diagnostic is emitted (also
implicates INV-3). The only hyphen tests call `answer_enumeration` directly, never
through placement.

- **Repro:** `{"clues":[{"answer":"WELL-BEING"},{"answer":"BELOW"}]}` →
  `arrange --strict --size 15` → 14 letter-cells incl. `-`. "WELL BEING" (space) → 13
  cells; "WELL-BEING" → 14.
- **Fix:** strip `-` as well as `' '` when deriving placed letters at all sites
  (crossword.pl 270/337/381/443 and quality.pl `word_letters/3`). The emitter already
  carries the **original** answer atom (`placed_to_word`), so export's enumeration
  derivation keeps the hyphen.

### R2 — A `fill` seed word not in the dictionary makes a fillable grid fail
`fill.pl:139` (`fill_search`/`select_mrv`/`empty_slots`) · dimension=spec-conformance ·
AC-FILL-2, OD-3, INV-3 · **CONFIRMED**

Seeds are LOCKED as hard pins you "fill around" (§8.4/OD-3). But `apply_seed` only
binds the seeded slot's vars; `fill_search` still **selects** that slot, `candidates/4`
returns `[]` (the seed isn't in the dict), the 0-candidate slot sorts first, and the
search declares the user's own pinned slot unfillable — even when a legal completion
exists. This forces seeds (a setter's theme/own words) to also be dictionary words.

- **Repro:** seed `CAT` across row 0 of `fixtures/fill_grid_3.json` with a dict that
  omits `CAT` (`grep -vx CAT`) → `fill: no dictionary word fits slot(s) [1]`, exit 1.
  The same seed against the full sample wordlist succeeds.
- **Fix:** track seeded slots and exempt them from candidate selection and from
  `empty_slots` reporting (a seed pin is already satisfied). Keep requiring a dictionary
  candidate for every *non-seed* slot (preserves AC-FILL-1).

## Med

### R3 — emit→re-ingest-as-fragment round-trip is not byte-identical
`crossword.pl:634` (`add_clue_nums`) · dimension=spec-conformance · AC-EMIT-2 · **CONFIRMED**

When two words share a start cell, `add_clue_nums([_-[W1,W2]|...])` emits the
across/down pair in **placement order**, which is never canonicalized — and the strict
DFS vs. the fragment re-solve build `PlacedWords` in different orders. So emit →
re-ingest as `--fragment` diverges. `arrange.pl:277` advertises this as load-bearing
and `tests/arrange.plt:173` asserts byte-identity, but `bundled_17` happens never to
flip the pair, so the failing path is unexercised. (AC-EMIT-1/INV-2 are scoped to
identical input+flags and are *not* breached — `--fragment` differs from the original
invocation.)

- **Repro:** `arrange --strict --size 13 --size-mode fixed` on
  `fixtures/benchmark_08_words.pl`, re-ingest the emitted JSON as `--fragment` →
  `cmp` differs at byte 6325 (the start-1 across/down pair flips).
- **Fix:** canonicalize the `words` array at emit time — sort by `(number, direction)`
  (or each clue-number group across-before-down) so any placement order serializing the
  same crossword is byte-identical.

### R4 — `not_proven` (AC-ARR-1c) and the budget flag (AC-ARR-10) are untested
`tests/arrange.plt` · dimension=test-gap · AC-ARR-1, AC-ARR-10 · **CONFIRMED**

AC-ARR-1 demands three distinct outcomes (placed / infeasible-naming-words /
not-proven-within-budget); the split rests on one `memberchk(budget,Results)` ternary
(`arrange.pl:93`). `grep` for `not_proven`/`budget`/AC-ARR-10 across `tests/` returns
nothing; `arrange.plt` asserts only `placed` and `infeasible`, and the comment at
`tests/arrange.plt:68` ("(c)/(b)") misleadingly implies (c) is covered. A regression
mis-tagging budget exhaustion as `infeasible` — falsely naming an "unplaceable word",
which `design-spec.md:234` flags as correctness-critical — would pass green.

- **Fix:** force budget exhaustion (override `arrange_budget`/stub the inference limit)
  and assert `Outcome==not_proven`, distinct from an unplaceable-word report.

## Low

### R5 — `enum_segments` emits zero-length segments for stray separators
`export.pl:54` · dimension=spec-conformance · AC-ENUM-1, AC-EXP-3 · **CONFIRMED**

No input trim/normalize/reject, and `enum_segments` unconditionally appends the current
run length on every separator and at end-of-list. A typo like `"TT "` (trailing space)
yields enumeration `(2,0)`; `"ROCK  ROLL"` → `(4,0,4)`; `" AB"` → `(0,2)`; `"RE-"` →
`(2-0)`. Legitimate inputs render correctly — this is a missing normalization step.

- **Fix:** only emit a run length when `N>0` and drop a dangling trailing separator; or
  trim/collapse separator runs (or reject leading/trailing/adjacent separators) at input
  validation.

### R6 — `select_mrv` recovers the chosen slot by start cell only
`fill.pl:149` · dimension=correctness · INV-2 (not breached) · **CONFIRMED**

When a start cell begins both an across and a down slot, the sort key is `c(Count,Start)`
but recovery is `Best=slot(BestStart,_,_,_)` via `select/3` — returning the first
start-matching slot (across, since `fill_grid` generates across before down) regardless
of which direction held the min count. The comment at `fill.pl:155` even claims recovery
"by its unique start+dir" while the code uses start only. Output stays deterministic and
complete (the other slot is reached on backtracking), so INV-2 is **not** violated; the
impact is a heuristic regression that could, in pathological grids, drive search to the
budget.

- **Fix:** carry direction in the sort key (`c(Count,Start,Dir)`) and recover on
  `slot(BestStart,BestDir,_,_)`; fix the contradictory comment.

### R7 — strict arrange's 4-corner sweep does not share the inference budget
`arrange.pl:71` · dimension=footgun · AC-ARR-1c, §7.3 · **CONFIRMED**

`arrange_best_layout` iterates all 4 start corners, each calling `construct_one` with
the **same** full `arrange_budget(500_000_000)` and no shared cap/early-exit, so a
hard/infeasible input can pay up to 4×500M ≈ 2B inferences. The outcome is
spec-compliant (AC-ARR-1c; §7.3 mandates an inference-count budget for determinism, with
a wall-clock cap only optional), so this is a UX/latency wart, not a defect.

- **Repro:** `arrange --strict --size 15 --input fixtures/toc_demo.pl` runs ~100s wall
  before the correct exit 1 (`not proven within budget`). (At the fixture's intended
  size 25 it is ~90ms.)
- **Fix:** share/decrement one inference budget across the sweep (and/or short-circuit);
  optionally add the §7.3-permitted documented wall-clock safety valve marking output
  `truncated`.

### R8 — spec-mandated `--check-target` (§7.2 MUST) is unimplemented; the LOCKED spec self-contradicts
`crosswordsmith:93` (`arrange_opts_spec`); `arrange.pl:29` (`check_target/2`) ·
dimension=spec-conformance · §7.2, §7.5/DP-1, OD-9 · **CONFIRMED**

`arrange --check-target N` is rejected ("could not parse options", exit 1) — no such
opt exists, and the cap target is hardcoded `check_target(L,T):-T is (L+1)//2`. Yet
§7.2:228 ("MUST be a real tunable") and §7.5/DP-1:254 ("the tunable escape hatch where
ceil(L/2) is unreachable") mandate it, while §5's CLI flag enumeration **omits** it. So
the implementation conforms to §5 but contradicts §7.2/§7.5. Practical impact is minimal
(DP-1 says the cap is largely inert on realistic inputs; the default is locked-correct).

- **Decision needed (author):** either add `--check-target N` to `arrange_opts_spec`,
  thread it into `check_target`/`word_reward`, and list it in §5; **or** move it to §3
  Non-goals so the LOCKED spec stops mandating an unreachable flag. This is a scope
  decision, not a mechanical fix.

### R9 — optparse auto-generates undocumented `--no-<flag>` negations
`crosswordsmith:97` · dimension=footgun · AC-CLI-2 · **CONFIRMED**

Per `optparse.md:36`, every `type(boolean)` opt across the four `*_opts_spec` blocks
gets an auto-negation form (`--no-strict`, `--no-best-effort`, `--no-enumerate`,
`--no-help`, `--no-allow-asymmetry`); none appear in `--help`, contradicting AC-CLI-2
("no undocumented flag exists"). Worse, `--no-strict` sets `strict(false)`, and
`drop_contract/2` then falls through to `Drop=strict` — so the user gets strict
behaviour from a flag that reads as its opposite, silently (exit 0).

- **Fix:** disable/validate negation (whitelist allowed flags via `opt_parse`
  `ParseOptions`, or reject unknown forms), or document the negation forms.

### R10 — stock-grid schema's documented `symmetry` field is never parsed/validated
`stockgrid.pl:21` · dimension=contract · AC-LINT-4, §8.3 · **CONFIRMED**

`stockgrid_load` destructures only `name`/`size`/`mask`; the §8.3 schema field
`{name,size,symmetry,mask}` is dropped (and `stockgrid.pl:143`'s error message also omits
it, contradicting the file's own line-5 header). Validation re-derives `rot180` from the
mask independently, so the declared `symmetry` value is never cross-checked — a garbage
string (or its absence) validates PASS. Contract/doc drift, not a legality bug (a truly
asymmetric grid still FAILs lint's symmetry rule).

- **Fix (low-friction):** drop `symmetry` from the §8.3 schema and the file header (mask
  is the single source of truth). Alternatively, assert declared symmetry == derived
  `rot180` and warn/fail on mismatch.

### R11 — `fill` `not_proven` (AC-FILL-1) and re-validation (AC-FILL-4) untested
`tests/fill.plt` · dimension=test-gap · AC-FILL-1, AC-FILL-4 · **CONFIRMED**

`fill.plt` has no `lint`/`lint_run` call and exercises only `filled` and `infeasible`;
the only lint golden lints an *arrange* output, never a fill output, so no test
re-validates a produced fill against the metric/lint predicates (AC-FILL-4). And
`not_proven` (`fill.pl:183-184,212-215`) is reachable (a tiny inference limit yields
`inference_limit_exceeded`) but untested. AC-FILL-4 risk is low (fill only assigns
letters into a pre-validated mask and can't structurally degrade the layout); the
untested `not_proven` branch is the substantive gap.

- **Fix:** add a budget-exhaustion fill test asserting `Outcome==not_proven`, and a test
  that lints a fill output under a fixed profile and asserts the structural verdict is
  unchanged/PASS.

### R12 — `AC-LINT-2` FAIL→non-zero-exit mapping has no golden/test
`run_tests.sh`; `crosswordsmith:253` · dimension=test-gap · AC-LINT-2 · **CONFIRMED**

The single lint golden uses `--profile toc`, which is advisory-only (every rule `warn`)
and can never FAIL (always exit 0), and `check_golden` captures only stdout — discarding
the exit code; `lint.plt` calls `lint_run/5` directly, not the CLI. **Mutation proof:**
editing `crosswordsmith:253` to `-> true` (so FAIL exits 0, breaking AC-LINT-2) leaves
the full suite green. Production mapping itself works.

- **Fix:** add a golden/CLI test that lints a known-failing layout under `blocked-uk` and
  asserts non-zero exit, or have `check_golden` also compare exit status.

### R13 — `AC-EXP-2` Exolve round-trip asserted only as static line structure
`tests/export.plt` · dimension=test-gap · AC-EXP-2 · **CONFIRMED**

AC-EXP-2 claims the Exolve output "round-trips through Exet (load→save→equivalent
grid+entries)". The only automated coverage (`test(exolve_structure)`) merely
`memberchk`s a few fixed strings; there is no inverse Exolve parser in the repo, so no
equivalence is ever computed. A byte-exact golden guards against regression but does not
establish Exet ingestibility, and nothing records that the manual Exet check was done.

- **Fix:** align the spec to state AC-EXP-2 is verified manually / by golden pin (an
  in-program emit-then-reparse would only prove self-consistency, not third-party Exet
  ingestion).

### R14 — `AGENTS.md` describes the retired `crossword.pl` CLI + a dead golden
`AGENTS.md:26` · dimension=doc-drift · **CONFIRMED**

`AGENTS.md:26` calls `crossword.pl` "the executable script and main implementation",
`:29` names a single `tests/crossword.plt` as "the" coverage (now 6 `.plt` files), and
`:30-31` cite `tests/golden/grid_17_topleft_across.txt` as the golden for
`./crossword.pl ... 17 topleft_across`. That golden does not exist, and the command now
prints a migration hint and exits 1 (`crossword.pl` was library-ized; the CLI is
`crosswordsmith`). `README.md:300` correctly contradicts this.

- **Fix:** rewrite the Project Shape section to point at the `crosswordsmith` CLI and the
  six `tests/*.plt` files; drop the dead golden reference and command; re-check the
  `make golden`/`update-golden` notes against the new fixture set.

### R15 — `STATUS.md` "system of record" reports stale aggregate test counts
`docs/STATUS.md:84` · dimension=doc-drift · **CONFIRMED**

Line 84's aggregate says "full suite 118/118 plunit + 4 goldens green", while
`./run_tests.sh` reports **152 passed** and **8** golden OK lines. The per-suite bullets
just below (85–88) already enumerate lint/export/stockgrid/fill correctly, so the added
coverage isn't hidden — only the headline aggregate was not re-summed.

- **Fix:** update to 152 plunit + 8 goldens (or stop hard-coding the count).

## Nit

### R16 — design-spec §6.5 lists word fields the emitter never produces
`crossword.pl:740` (`placed_to_word`); `docs/design-spec.md` §6.5 · dimension=doc-drift · **CONFIRMED**

§6.5 writes the word object as `{answer, cells, dir, num, len, start, meta}`, but
`placed_to_word/4` emits `{number, direction, answer, cells, meta}` — no `len`/`start`,
and `dir`/`num` are the *internal* placed-word field names. The authoritative JSON
contract (`docs/json-output-spec.md` §6.2) already matches the code, so consumers code to
a correct contract; this is a §6.5 summary-line inconsistency only.

- **Fix:** edit §6.5 to read `{number, direction, answer, cells, meta}` (matching
  `json-output-spec.md` and the emitter). Do **not** add `len`/`start` to the emitter.

### L1 — `UKACD18` was mislabelled "BSD-3" (med, doc)
The spec (§3, §8.4, OD-2) and README called the default lexicon "UKACD18 (BSD-3)".
It is **not** BSD-3: UKACD is *"Copyright © J Ross Beresford 1993–2000. All Rights
Reserved"*, a **custom freeware license** — freely copyable/redistributable, but on
the condition that *"the copyright notice must be prominently displayed and the text
of this document … included verbatim."* No current AC-X-4 violation (only the trivial
sample ships, not UKACD18). **Fixed:** corrected the label in §3/§8.4/OD-2 + README to
"redistributable freeware (ship its copyright + license verbatim); not BSD-3."
**Obligation when bundled:** ship Beresford's notice + license text verbatim alongside
the wordlist.

### L2 — vendored SWI-Prolog manual is CC BY-SA 3.0, not surfaced (low, doc)
`docs/reference/swi-manual/**` is the SWI-Prolog reference manual, **CC BY-SA 3.0
Unported** (attribution + *share-alike* copyleft) — confirmed against the
version-matched installed docs and swi-prolog.org. The CC BY-SA notice **does**
survive in `swi-manual/index.md` (so attribution is present), but (a) it is buried,
not surfaced where a redistributor looks, and (b) CC BY-SA is copyleft, not "permissive"
in §3's strict sense. The repo's top-level `LICENSE` is MIT, which does not cover this
subtree. **Fixed:** added a "License (third-party)" note to `docs/reference/README.md`

share-alike scoped to the manual copy, not the rest of the repo). Functionally a

### Residual decision (author)
§3 says "ship only permissive data," but the chosen default lexicon (UKACD18) is a
*custom freeware* license and the vendored manual is *copyleft* — neither is
"permissive" in the strict OSI sense, though both are freely redistributable with
attribution. Either (a) read §3 as "freely-redistributable with attribution" (now the
working wording — UKACD18 stays the default, shipped with its verbatim notice), or
(b) hold to strictly-permissive and pick an MIT/CC0 default lexicon. (a) is the
low-friction reading and matches DP-2; recorded here so the choice is explicit.

## Determinism fuzz (INV-2 · AC-EMIT-1 · AC-X-2 · §5.2 — 2026-06-30)

Follow-up on the determinism coverage gap. Added `tests/determinism_fuzz.sh`
(`make fuzz`) — an on-demand harness (~22 s, **not** part of the fast `make
test`) that runs a matrix of **54 cases** spanning every verb, broad flag
combinations (`--strict`/`--best-effort`, both size-modes, `--candidates`,
`--enumerate`, `--check-target`, `--fragment`, `--flag=value` form, all four
lint profiles ± `--allow-asymmetry`, both export formats, fill ± seeds) and
**degenerate inputs** (empty clue list, single-letter, unicode, duplicate,
hyphen/space/punct/digit/lowercase answers, malformed JSON). Each case runs as
**three separate processes**; the harness asserts identical exit code + **byte-
identical stdout** across runs, a defined exit (0/1, never a hang/crash), and
the §5.2 `--out` partial-write contract.

**Result: INV-2 holds — 0 nondeterministic cases, 0 hangs, 0 ungraceful
crashes.** Every degenerate input either produces a valid layout or fails
cleanly (reported error, exit 1). The only thing the fuzz surfaced was a
distinction worth recording (no code defect):

- **§5.2 `--out` semantics differ by verb (by design).** `arrange`/`fill` exit
  non-zero on a genuine production failure and correctly leave **no** file (no
  partial write). `lint` always writes a **complete** report and uses its exit
  code as the *verdict* (AC-LINT-2), so a FAIL verdict legitimately pairs a
  complete `--out` file with exit 1 — not a "partial" file, so §5.2 is honoured.
  The harness encodes both contracts (`nofile_on_fail` vs `report_always`).

### V1 — Exet's Save crashes on a null title; crosswordsmith emitted none
`export.pl` (`layout_to_exolve/2`) · severity=low (interop) · AC-EXP-2 · **fixed**

crosswordsmith's Exolve emitted no `exolve-title`, so Exet set the title to `null`
and its `fileTitle()` (`title.replace(...)`) threw
`TypeError: Cannot read properties of null (reading 'replace')` in
`updateSavePanel`, silently breaking **Save**. (Arguably an Exet bug — the title is
"optional" — but the actionable fix is ours.) **Fix:** emit a default
`  exolve-title: Untitled` (an editable placeholder). Verified: the export re-ingests
cleanly and the golden gains exactly that one line; the human confirmed Save works
once a title is present. `tests/export.plt` asserts the title line.

**Sibling fix — ipuz export.** The **ipuz** export had the same gap (no `title`).
Exet imports ipuz by converting it to Exolve (`exolveFromIpuz`) and reuses the
same format-agnostic `fileTitle()` (`this.puz.title.replace(...)`), so a titleless
ipuz opened in Exet would crash Save identically — confirmed by reading Exet's
source (not re-tested in the browser). `title` is *optional* in ipuz, so kotwords
(the ipuz export's primary consumer, AC-EXP-1) is unaffected; but for parity and
Exet-robustness the ipuz export now emits `"title":"Untitled"` too (standard
optional field). `tests/export.plt` `ipuz_header` asserts it; golden updated.
