## How to read each finding

- **severity** — `high` (wrong output / crash on realistic input) > `med` (edge-case-wrong, real
  robustness gap, or a measurable hot-path cost) > `low` (latent / cosmetic-but-real) > `nit`
  (pure style).
- **category** — `A` predicate-use / correctness · `B` reinvented wheel (stdlib exists) ·
  `C` elegance / performance / best-practice.
- **behaviour** — `preserving` (cannot change observable output: the plunit suite, the CLI goldens,
  and the JSON/ipuz/exolve output contracts stay byte-identical) vs `risk` (a fix *could* change
  observable output and must be golden/diff-checked).
- **verdict** — `CONFIRMED` (reproduced with `swipl`, or directly code-grounded against the manual
  / library source) · `PLAUSIBLE` (well-grounded, not independently reproduced).
- **status** — remediation state: `open` / `in-progress` / `fixed` / `rejected` / `wontfix`.

## Method

Five review lanes ran in parallel — `crossword.pl`; `arrange.pl`; `fill.pl`+`stockgrid.pl`;
`lint.pl`+`quality.pl`; `export.pl`+harness — each with an identical rubric (correct predicate use /
reinvented wheels / elegance), each required to verify every non-trivial builtin/library call
against the on-disk manual or library source before flagging, and to distinguish real defects from
style. A synthesis pass deduped, ranked, and independently reproduced the one correctness finding
(P1) with `swipl`. Ground truth at audit time: full suite green (**168 plunit + 8 CLI goldens**).

## Med

### P1 — `is_end_cell(down,…)` off-by-one lets a down word merge collinearly
`crossword.pl:832` · category `A` · behaviour `risk` · verdict **CONFIRMED** · status **fixed**

```prolog
is_end_cell(down, Num, Length) :- Num >= (Length - 1) * Length.   % should be  >
```
`>=` misclassifies exactly one cell — the last cell of the second-to-last row (e.g. cell **20** on a
5×5) — as a down "end cell", even though the cell directly below it (**25**) is on-grid. That
predicate gates the maximal-run / no-merge invariant: in `check_next_cell/4` (`:538`) a down word
ending there skips the "cell below must be empty" guard, and in `placed_boundary_cell/3` (`:516`) its
trailing boundary cell is never protected by `no_word_merge/3`. Net effect: a down word can be placed
collinear-adjacent to existing content at cell `(L-1)*L`, producing exactly the "word inside a word"
merge the code documents as forbidden (`crossword.pl:491-506`).

- **Evidence:** reproduced with `swipl` — for `L=5`, current classifies cell 20 as `end`, but
  `next_cell(down,20,5)=25 ≤ 25` (a cell below exists). The sibling
  `is_start_cell(down, Num, Length) :- Num =< Length` is correct; this is the odd one out. Compare
  `fits_on_grid(down,…)` (`:824`) which allows a last letter up to `GridLen*GridLen`.
- **Fix:** `is_end_cell(down, Num, Length) :- Num > (Length - 1) * Length.` Add a focused plunit test
  placing a down word ending at cell `(L-1)*L` and asserting the adjacency guard rejects a collinear
  neighbour. **Golden/diff-check** the CLI outputs — this can legitimately change a placement.

### P2 — Broad `catch/3` reports genuine errors as "infeasible" (two engines)
`arrange.pl:139-148` & `arrange.pl:518-529`; `fill.pl:200-210` · category `A` · behaviour `preserving`
· verdict **CONFIRMED** · status **open**

Both engines wrap the search in `catch(call_with_inference_limit(...), _, <infeasible outcome>)`. But
`call_with_inference_limit/3` handles the budget **internally** — on hitting the limit it binds
`Result = inference_limit_exceeded` and *succeeds*; it only re-throws for *other* exceptions. So the
outer `catch` never fires for the budget path — it only ever catches real type/existence/resource
bugs, and silently reclassifies them as an ordinary "no full placement / no complete fill" result
(`arrange` → `exhausted`/`infeasible`; `fill` → `exhausted` → the "no complete fill exists" message).
A programming error is thus reported to the user as a normal infeasibility.

- **Evidence:** `reference/swi-manual/metacall.md:79-83` — on limit the goal "is aborted by injecting
  … `inference_limit_exceeded`" and "`Result` is unified with the atom `inference_limit_exceeded`";
  "If `Goal` throws an exception, `call_with_inference_limit/3` re-throws" it. The catch is therefore
  unnecessary for the limit and swallows everything else.
- **Fix:** narrow the recovery to expected exceptions and re-raise the rest, e.g.
  `catch(Goal, E, ( expected_infeasible(E) -> Recover ; throw(E) ))`. Same one-line change at all
  three sites. (If the broad catch is deliberate belt-and-braces, add a comment saying so — currently
  it reads as a bug.)

### P3 — MRV counting materializes full candidate word-lists just to `length/2` them
`fill.pl:159-172` (+`:127`) · category `C` · behaviour `preserving` · verdict **CONFIRMED** ·
status **open**

`select_mrv` maps `slot_candidate_count/4` over every unfilled slot at every search node; that helper
calls the full `candidates/4`, which builds the entire candidate word list via
`maplist(nth0_of(Words), Indices, Cands)` and then keeps only `length(Cands, Count)`. The word list is
pure waste during counting, and the winner's list is then rebuilt a second time (`fill.pl:165`).
Compounding it, `nth0_of` calls `nth0/3` with a bound index, which walks `Words` from the front each
time — O(n²) over an ascending index set.

- **Evidence:** `candidates/4` at `fill.pl:120-128` always maps indices→words before returning;
  `slot_candidate_count` (`:170-172`) discards the list. Bound-index `nth0/3` is a front scan:
  `/usr/lib/swi-prolog/library/lists.pl:264-274` (`'$seek_list'`/`nth0_det`).
- **Fix:** give counting an index-set-only path (`length(Indices, Count)`), and thread the winner's
  candidate list *out* of the count pass instead of recomputing it — or store word refs directly in
  the index (`k(Len,Pos,Char) → ordset-of-word-refs`) so no `nth0` mapping is needed. `needs-benchmark`
  (portable metric = INFERENCES, per `experiments.md`).

### P4 — Checked-bitmap metric recomputed ~4×/word, and lives in the wrong module
`lint.pl:191` + `quality.pl:204,210` · category `C` (+DRY) · behaviour `preserving` ·
verdict **CONFIRMED** · status **open**

`word_checked_bitmap/3` (lint) hand-computes per-cell checkedness with the same core inside
`word_checked_count/3` (`quality.pl:204`) and `word_max_unch_run/3` (`quality.pl:210`). During a lint
run the toc profile evaluates checked-half + max-unch-run + double-unch-ends + odd/even per word, so
`dir_cells/3` — a `findall` over *all* placed words then `sort/2` (`quality.pl:190`) — is recomputed
~4× per word: roughly O(N²·log). It is also an architecture smell: this is metric logic, but
`quality.pl` is meant to be *the* shared-metrics module (`quality.pl:6-12`), and `lint.pl` should
delegate.

- **Evidence:** `quality.pl:190-195` (`dir_cells` = `findall`+`sort` over all `Placed`); callers at
  `lint.pl:193,194`, `quality.pl:206,212`. `aggregate_all(count,…)` semantics confirmed at
  `reference/swi-manual/aggregate.md:32,66-69`.
- **Fix:** compute `dir_cells(Placed,across,AC)` and `dir_cells(Placed,down,DC)` **once** in `lint_run`
  (`lint.pl:100`) and thread the perpendicular set into the evaluators; hoist one canonical
  `word_checked_bitmap/…` into `quality.pl` and derive count (`sum_list`/`aggregate_all(count)`) and
  max-run (a fold over the bits) from it. Small at lint's scale (dozens of words) but removes the
  duplication and the redundant sorts.

## Low

### P5 — `answer_meta/3` linear rescan makes emit O(n²)
`crossword.pl:766-771` · category `C` · behaviour `preserving` · verdict **CONFIRMED** · status **open**

`answer_meta/3` does a `member/2` scan of the full input `Words` list, and `build_words/4` calls it
once per placed word (`:733-735` → `placed_to_word/4:759`), so the metadata rejoin is O(n²). `n` is
small for crosswords, so this is minor.
**Fix:** build an answer→meta assoc once (`list_to_assoc`/`get_assoc`) and thread it into
`placed_to_word/4`, making the join O(n log n).

### P6 — `intersection/3` builds the whole list just to test non-emptiness
`arrange.pl:194` · category `C` · behaviour `preserving` · verdict **CONFIRMED** · status **open**

`word_shares_letter/2` does `intersection(Ls, OLs, [_|_])`, computing the full intersection only to
check "≥1 shared letter". Runs on the rare infeasible-report path, so negligible, but a short-circuit
is cheaper and clearer.
- **Evidence:** `/usr/lib/swi-prolog/library/lists.pl:793-796` — "The complexity of this predicate is
  |Set1|*|Set2|".
- **Fix:** `( member(L, Ls), memberchk(L, OLs) -> true ; fail )`.

### P7 — `length/2` walks the whole run to compare against 2
`stockgrid.pl:81` · category `C` · behaviour `preserving` · verdict **CONFIRMED** · status **open**

`line_lights` does `length(Run, L), L >= 2`, an O(run) traversal for an O(1) test.
**Fix:** `Run = [_,_|_]`.

### P8 — Inlined letter-normalization duplicates the existing `entry_letters/2`
`crossword.pl:269-271, 337-339, 383-385` · category `B` · behaviour `preserving` · verdict **CONFIRMED**
· status **open**

The three-line `atom_chars(Word,L0), delete(L0,' ',L1), delete(L1,'-',Letters)` is inlined at three
sites (`assign_words/9`, `mrv_count/8`, `assign_words_inc/9`) even though `entry_letters/2`
(`crossword.pl:446-449`) already encapsulates exactly this, and all three call sites already hold an
`[Word|_]` entry.
**Fix:** replace each inlined block with `entry_letters(Entry, Letters)`. (Note: this is the same
hyphen-stripping path touched by revamp R1 — keep that fix's semantics.)

### P9 — ipuz and exolve clue collection are byte-for-byte duplicated
`export.pl:118-128` & `export.pl:168-179` · category `B` · behaviour `preserving` · verdict **CONFIRMED**
· status **open**

`ipuz_clues/3` and `exolve_clue_lines/3` are the same `findall(Num-X, (member(W,Words),
word_dir(W,Dir), <mk>(W,Num,X)), Pairs) → keysort → pairs_values` skeleton, differing only in the
per-item goal; likewise `ipuz_clue/3` and `exolve_clue_line/3`.
**Fix (behaviour-preserving):**
```prolog
:- meta_predicate collect_by_number(+, +, 3, -).
collect_by_number(Words, Dir, MkItem, Items) :-
    findall(Num-Item,
            ( member(W, Words), word_dir(W, Dir), call(MkItem, W, Num, Item) ),
            Pairs),
    keysort(Pairs, Sorted),
    pairs_values(Sorted, Items).
```
then `collect_by_number(Words, across, ipuz_clue, Across)`, etc. Do **not** change the emitted shape —
it is golden-tested and spec-pinned (`json-output-spec.md`, `exet-verification.md`).

### P10 — `checked_half` re-derives the `ceil(L/2)` threshold `word_meets_half/2` already owns
`lint.pl:148-151` · category `B` · behaviour `preserving` · verdict **CONFIRMED** · status **open**

The shared metric `word_meets_half/2` (`quality.pl:198`, `CC >= (L+1)//2`) already defines
"half-checked", but the lint rule inlines `T is (L+1)//2, C >= T` because it needs `C` and `T` for the
detail string. The "half" definition now lives in two files.
**Fix:** expose the threshold once (e.g. `word_half_threshold(L,T)` in `quality.pl`, or have
`word_meets_half` also yield `CC`/`T`) and have the rule reuse it.

### P11 — `quality.pl` uses `library(pairs)` predicates without importing it
`quality.pl:25` (missing import; uses at `:35,:37,:160`) · category `A` · behaviour `preserving` ·
verdict **CONFIRMED** · status **open**

`map_list_to_pairs/3`, `pairs_values/2`, `pairs_keys_values/3` all live in `library(pairs)`
(`/usr/lib/swi-prolog/library/pairs.pl:38-43,180`) but `quality.pl` imports only `ordsets` and
`apply`. It works today via autoload, but the sibling `fill.pl:20` imports `library(pairs)` explicitly
(AGENTS.md: "prefer the existing style"), and autoload reliance breaks under a standalone
`qsave_program(..., [autoload(false)])`.
**Fix:** add `:- use_module(library(pairs)).`

### P17 — Optional: silent-fail boundaries could fail loud
`export.pl:114-115, 165-166`; `crossword.pl:629-642`; `arrange.pl:33-36` · category `A` ·
behaviour `preserving` · verdict **CONFIRMED** · status **open**

Three spots fail silently on out-of-contract input (all safe under current invariants — optional
hardening, not live bugs):
- `export.pl` `ipuz_solution_cell`/`exolve_cell_char` read `get_dict(letter, Cell, _)` with no
  fallback, and `ipuz_clue`/`exolve_clue_line` require `answer`; a malformed cell makes
  `layout_to_ipuz/2` *fail* rather than throw like `export_load/2` does. Asymmetric with
  `ipuz_puzzle_cell` (`:108`), which guards `number`.
- `crossword.pl` `add_clue_nums/3` (`:629`) matches only groups of 1–2 words; a >2 group (invariant
  says impossible) would fail the whole `assign_clue_numbers/2` silently.
- `arrange.pl` `check_target/2` (`:33`) assumes at most one `check_target_override/1` fact; the `->`
  commits to the first if the CLI ever asserts twice without `retractall`.

**Fix (optional):** add `must_be`/throw at these boundaries for parity with `export_load`'s
validation, or document the invariant inline (esp. the `:- dynamic check_target_override/1` line).

### P12 — Missing `meta_predicate` decls on goal-taking helpers
`crossword.pl:112-117, 352-353` · category `C` · behaviour `preserving` · status **open**

`with_output/2` and `capped/2` accept and `call/1` a Goal but have no `:- meta_predicate`. Harmless

### P13 — Spurious choicepoint from `select/3` in `select_mrv`

`select(Best, Slots, Rest)` with a unique `Start+Dir` still leaves a choicepoint, explored and failed
on every backtrack.
**Fix:** `once(select(Best, Slots, Rest))`.

### P14 — Redundant unification in `candidates/4`
`fill.pl:123` · category `C` · behaviour `preserving` · status **open**

`findall(P-Ch, ( nth0(P, Vars, V), nonvar(V), Ch = V ), Bound)` — the `Ch = V` is superfluous.
**Fix:** `findall(P-C, ( nth0(P, Vars, C), nonvar(C) ), Bound)`.

### P15 — Redundant trailing `!` after an already-deterministic if-then-else
`crossword.pl:589, 604` · category `C` · behaviour `preserving` · status **open**

In `adj_is_free/4` the body is `( C1 -> A1 ; C2 -> A2 ; A3 ), !` where every arm is semidet, so the

**Fix:** drop the trailing `!` in `adj_is_free/4`.

### P16 — Redundant `list_to_ord_set/2` on an already-ordered set
`lint.pl:219, 244` · category `C` · behaviour `preserving` · status **open**

`connected/2` and `symmetry_deficit/3` call `list_to_ord_set(Filled, …)`, but `Filled` comes from
`filled_cells/2` via `sort/2` (`lint.pl:96`) and is already a dedup'd ordset at every call site.
**Fix:** use `Filled` directly.

## Appendix A — Verified correct · do NOT re-flag

Each of these was checked against the manual / library source and is deliberate and idiomatic. A
future audit should not re-open them without new evidence.

- **`map_list_to_pairs` (not `findall`) to preserve term identity** for the `==`-based `remove_x/3`
  during backtracking search (`crossword.pl`, comments at `:402-404,:743-744`). Switching to `findall`
  would `copy_term` and break the search.
- **`intersection/3` + `list_to_set/2` at `crossword.pl:468-469`** is *not* redundant: SWI's
  `intersection/3` retains duplicates from Set1 (`lists.pl:804`), so the dedup is needed before
  enumerating crossing letters.
- **`sort/4` stability + duplicate-keeping for deterministic tie-breaks** (`arrange.pl:106,218,659`;
  `quality.pl:107,124`) — sorts on the key only so the large value term is never compared; stable
  order preserves corner/seed generation order (INV-2). Verified `builtinlist.md:68-74`.
- **`foldl` + yall lambda instead of `aggregate_all(sum)`** in `layout_reward/4` (`arrange.pl`) — the
  *safer* choice: `aggregate_all(sum)` would over-count if `word_checked_count` ever left a
  choicepoint.
- **findall-then-wire-shared-variables** in `fill.pl:33-57` — correct; the comment explains why both
  `findall` and a yall lambda would `copy_term` and break the shared crossing variables.
- **`split_runs/3`** (`stockgrid.pl:83-92`) — genuinely custom (no stdlib "split by membership"
  exists — `clumped`/`partition` don't fit); correct, deterministic, clean first-arg indexing. Leave
  as-is (a DCG rewrite is not clearer).
- **`reverse(Bits,[0,0|_])` end-detection** (`lint.pl:197`) — correct: `reverse` puts the last cell
  first, so `[0,0|_]` means the final two cells are unchecked. `Bits=[0,0|_]` handles the start.
- **`barred_max_unch/2` lookup table** (`lint.pl:182-188`) — intentional, primary-sourced, total over
  valid lengths, deterministic via first-arg indexing + cuts.
- **export JSON null/string handling** — `json_read_dict/2` defaults (`null`→atom `null`,
  strings→SWI strings; `library/ext/json/json.pl:139-140`) mean `white_cell/1 :- is_dict/1` correctly
  treats a black square as non-white. The enumeration tokenizer is bespoke, not a reinvented stdlib.
- **`~0f` with integer inference counts** in the bench harness — fine; `f` accepts an arithmetic
  expression and `~0f` prints an integer (`format.md:34,64-65`).
- **`aggregate_all(count, …)`** (`lint.pl:245`, `quality.pl:207`) and **`sort(0,@=<,…)`** MRV min
  (`fill.pl:161`) — correct templates, semidet, deterministic tie-break.
